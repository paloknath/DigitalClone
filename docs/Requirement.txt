Technical Specification: Teams Audio Agent (Audio-Only)1. High-Level ArchitectureThe agent acts as a "Virtual Participant" in Microsoft Teams. It utilizes a Browser Bridge to maintain the session and Virtual Audio Routing to connect the browser's media stream to the Pipecat pipeline.Host: Windows VPS (4+ vCPU, 16GB RAM recommended).Orchestration: Pipecat (Python).Intelligence: Gemini 2.5 Flash Live (via WebSocket).Transport: LocalAudioTransport (mapping to Windows Audio Devices).2. Infrastructure & Audio RoutingTo bridge the gap between the headless browser and Python, we use a dual-cable loopback system:Cable-A (Bot Mouth): Pipecat Output $\rightarrow$ VB-Cable A Input $\rightarrow$ VB-Cable A Output $\rightarrow$ Teams Mic.Cable-B (Bot Ears): Teams Speaker $\rightarrow$ VB-Cable B Input $\rightarrow$ VB-Cable B Output $\rightarrow$ Pipecat Input.Config Note: All virtual devices must be set to 48kHz / 16-bit in the Windows Sound Control Panel to prevent resampling latency and "crackling" audio.

Operational Guardrails
Process Priority: On Windows, the Pipecat process should be set to High priority to ensure the 20ms audio chunks aren't delayed by background OS tasks.

Echo Cancellation: Since Teams has built-in Echo Cancellation (AEC), disable Pipecat's software VAD/AEC initially to let Teams handle the raw stream, then enable SileroVAD in the pipeline if the agent becomes too "chatty."
